Privacy Policy

This privacy policy explains how CliniClarity processes, protects, and minimizes the use of medical data. We designed this system using privacy-first and security-focused principles aligned with healthcare data protection standards such as HIPAA.

🔍 Privacy Summary (Quick View)

📄 What data is collected

Users may upload medical reports for analysis. Authentication data such as user ID or email may be collected through Firebase, but it is kept separate from medical content.

🧼 How data is protected

Sensitive data is automatically removed using redaction before any AI processing. This ensures personal identifiers are not exposed to external systems.

⚡ How data is used

Only the minimum necessary portions of the document are used for processing through retrieval techniques. This reduces unnecessary data exposure.

☁️ Third-party services

External AI and database services are used, but only receive redacted and minimized data rather than full documents.

🗑️ Data storage

Medical reports are not permanently stored. Data is processed temporarily and cleared after the session.

🛡️ Security checks

Multiple validation layers ensure only valid documents are processed and AI outputs remain accurate and safe.

Detailed Privacy Policy

1. Data Collection

CliniClarity collects only the information necessary to provide its functionality. Users may voluntarily upload medical reports for analysis. When authentication is enabled, basic account information such as a user ID or email address is managed through Firebase Authentication. This authentication data is stored separately from medical report content and is not directly linked to the processed documents. This separation reduces the risk of associating personal identity with sensitive medical information and supports privacy-by-design principles.

2. Input Validation and Safe Processing

Before any processing begins, uploaded files are validated to ensure they meet expected requirements. This includes verifying that the file is a valid PDF and confirming that the content represents a medical report. Validation uses both rule-based checks and AI-assisted classification to prevent misuse, such as uploading malicious or unrelated files. By rejecting invalid inputs early, the system reduces its attack surface (the number of ways a system can be exploited) and improves overall security.

3. Data Redaction (Removal of Sensitive Information)

All uploaded reports undergo an automatic redaction process before any external processing occurs. Redaction means identifying and removing sensitive information such as names, identifiers, or other personal details. This is performed using Microsoft Presidio, a tool specifically designed to detect and anonymize personally identifiable information (PII) and protected health information (PHI). By removing sensitive data early in the pipeline, the system ensures that external services only receive anonymized content, reducing the risk of data exposure and supporting the HIPAA principle of confidentiality.

4. Data Minimization and Retrieval-Based Processing

CliniClarity uses a retrieval-based approach (often referred to as RAG, or Retrieval-Augmented Generation). Instead of processing an entire document at once, the system breaks the report into smaller segments and retrieves only the most relevant portions needed for a task. This means that the AI model works with limited, focused information rather than full reports. This approach reduces unnecessary data exposure and aligns with the HIPAA “minimum necessary” principle, which requires that only the least amount of data needed for a task should be used.

5. Temporary Caching

The system uses temporary caching during active sessions to improve performance and reduce repeated processing. Caching means storing results for a short period so the system does not need to repeat the same computation. This reduces the number of times sensitive data must be transmitted to external services. Cached data is short-lived, scoped to the user’s session, and not permanently stored. This design reduces exposure while maintaining efficiency.

6. Third-Party Services

CliniClarity uses third-party services to support functionality, including AI models for summarization and vector databases for retrieval. These services are accessed over secure HTTPS connections. Importantly, only redacted and minimized data is sent to these services rather than full, raw reports. While these providers maintain their own security measures, CliniClarity minimizes risk by limiting the type and amount of data shared.

7. Security Measures

CliniClarity implements multiple layers of security to protect user data. These include input validation, automatic redaction of sensitive data, restricted AI behavior, hallucination detection (ensuring outputs are based on real data), and protection against prompt injection (attempts to manipulate AI behavior through malicious input). Together, these safeguards reduce the likelihood of incorrect outputs, unauthorized data exposure, and system misuse. This layered approach supports both data integrity (ensuring correctness) and confidentiality.

8. Data Retention

CliniClarity does not store uploaded medical reports in a persistent database. Data is processed temporarily during a user session and is cleared afterward. Any temporary storage used for processing, such as caching or retrieval indexing, is short-lived and reset between sessions. This approach limits long-term exposure and reduces the impact of potential data compromise.

9. Authentication and Identity Protection

CliniClarity uses Firebase Authentication to manage user identity, sign in and sign up process securely. Authentication data is intentionally kept separate from uploaded medical reports. This means that even if authentication credentials were compromised, they would not directly expose sensitive medical content. This separation reduces linkage risk and enhances overall privacy protection.

10. HIPAA Alignment and Compliance

CliniClarity is designed to align with key HIPAA principles, including confidentiality, integrity, and the minimum necessary use of data. Confidentiality is supported through redaction and data minimization, ensuring sensitive information is protected. Integrity is supported through validation and output checking, ensuring that generated results are accurate and grounded in real data. The minimum necessary principle is achieved by limiting the amount of data processed at each step.

However, CliniClarity is not fully HIPAA compliant. Full compliance requires additional administrative, legal, and organizational measures beyond technical implementation. These include Business Associate Agreements (BAAs) with service providers and incident response procedures, and documented security policies. While this system demonstrates strong technical safeguards, these additional requirements are necessary for full regulatory compliance.
